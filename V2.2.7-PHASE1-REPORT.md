## Root Cause: v2.2.6 Windows Version Mismatch

**Confirmed: Hypothesis 2 (stale artifacts uploaded by MCP) + structural pipeline gap.**

### What happened

The `anvil_build` MCP tool (`/mcp-servers/anvil-release/lib/tools/build.js`) only builds the
native `aarch64-apple-darwin` binary. It does not build the other 4 targets. Those are built
by `release.sh` via Docker cross-compilation and land in `target/release-artifacts/`.

When `anvil_github_release` runs, it uploads **every file it finds** in `target/release-artifacts/`
without checking whether each artifact's embedded version matches the release tag.

For v2.2.6, `anvil_build` was called (rebuilding only the ARM Mac binary), but `release.sh`
had not been run fresh. The 4 cross-compiled artifacts in the artifacts dir were left over
from v2.2.1 (built 2026-04-14). The upload pushed all 5, with 4 stale.

### Evidence

Artifact timestamps and embedded versions before this fix:

| Binary | Embedded Version | Timestamp |
|---|---|---|
| anvil-aarch64-apple-darwin | 2.2.5 | Apr 19 21:36 |
| anvil-x86_64-apple-darwin | 2.2.1 | Apr 14 14:32 |
| anvil-x86_64-unknown-linux-gnu | 2.2.5 | Apr 14 14:35 |
| anvil-aarch64-unknown-linux-gnu | 2.2.1 | Apr 14 14:35 |
| anvil-x86_64-pc-windows-gnu.exe | 2.2.1 | Apr 14 14:35 |

The ARM Mac binary was 2.2.5 (not 2.2.6) because a build ran between 2.2.5 and 2.2.6 that
only rebuilt that one. All cross-compiled targets were from 2.2.1.
