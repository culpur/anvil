# Apache `flushpackets=on` for AnvilHub Build SSE Stream — Spec (#740)

**Tasks:** #674 (Build page + micro-service), #740 (this — Apache buffering fix)
**Target hosts:** guard.armored.ninja (CT 101), guard2.culpur.net (CT 110)
**Change owner:** Apache edge profile, `profile::apache_edge`
**Driver:** AnvilHub `/build` page streams generated spec tokens to the
browser via Server-Sent Events from the `anvil-mcp-builder` micro-service
on dev0001 (`10.0.70.80:4090`). Without `flushpackets=on` the default
mod_proxy_http behaviour buffers the response and breaks live token
streaming.

---

## 1. Why

`POST /api/builder/spec` with `Accept: text/event-stream` emits one SSE
event per generated token (`event: token` / `data: {...}`). The
`BuildClient.tsx` wizard updates the streaming token buffer reactively as
each event arrives. With Apache's default mod_proxy_http behaviour the
response is buffered until either the backend closes the connection or a
size threshold is reached — the user sees a long pause followed by a wall
of text, not a stream.

`flushpackets=on` (added to mod_proxy in Apache 2.4.7) instructs
mod_proxy_http to forward each chunk to the client as soon as it arrives
from the backend, with no intermediate buffering. This is the canonical
fix for SSE behind Apache reverse proxy.

The spec doc `v2.2.19-anvilhub-build-page-spec.md` Section 7 Open Question
#5 already flagged this requirement; #740 is the implementation.

---

## 2. Current state

The `api.culpur.net` vhost (or whichever vhost owns the `/api/builder`
path — the spec doc Section 5 specifies `api.culpur.net`) proxies to the
micro-service with a plain ProxyPass directive of the form:

```apache
# anvil-mcp-builder (dev0001:4090) — CURRENT (buffered, breaks SSE)
ProxyPass        /api/builder http://10.0.70.80:4090/api/builder
ProxyPassReverse /api/builder http://10.0.70.80:4090/api/builder
```

Apache 2.4.7+ defaults `flushpackets` to `off`, which buffers proxied
responses. Anvilhub's existing token-streaming UIs (Build wizard) require
the `on` setting.

---

## 3. Change

In the `api.culpur.net` Apache vhost (the file that fronts
`/api/builder/*` and forward-auths to Authentik), update the SSE ProxyPass
line to add `flushpackets=on` after the URL:

```apache
# anvil-mcp-builder (dev0001:4090) — SSE-compatible
ProxyPass        /api/builder/spec http://10.0.70.80:4090/api/builder/spec flushpackets=on
ProxyPassReverse /api/builder/spec http://10.0.70.80:4090/api/builder/spec

# Non-streaming endpoints — keep the default
ProxyPass        /api/builder http://10.0.70.80:4090/api/builder
ProxyPassReverse /api/builder http://10.0.70.80:4090/api/builder
```

Notes:

1. **Order matters.** Apache evaluates ProxyPass directives top-to-bottom
   and uses the first match. The `/spec` line MUST appear BEFORE the
   broader `/api/builder` line.
2. `flushpackets=on` is a per-directive flag on the source URL pattern;
   it must come AFTER the backend URL, separated by a space.
3. The corresponding `ProxyPassReverse` does NOT take `flushpackets` —
   that flag is mod_proxy_http-only and only meaningful on the forward
   path.
4. Generate (`POST /api/builder/generate`) and sandbox
   (`POST /api/builder/sandbox`) return single JSON responses — they
   benefit from the default buffered path. Only `/spec` needs the flush.

### Alternative — `Header set X-Accel-Buffering "no"`

The micro-service could set `X-Accel-Buffering: no` on the SSE response
to tell upstream proxies not to buffer. This works for nginx and some CDN
edges but mod_proxy_http on Apache ignores it. `flushpackets=on` is the
canonical Apache lever and is the chosen fix.

## 4. Exact path to edit

The vhost source file lives under the Puppet-managed Apache edge profile:

```
puppet-control/environments/production/site-modules/profile/files/apache_edge/vhosts/api.culpur.net.conf
```

The profile class `profile::apache_edge` (manifests/apache_edge.pp) ships
each vhost source file out to `/etc/apache2/sites-available/<name>.conf`
on guard and guard2 via Puppet's `file` resource with
`puppet:///modules/profile/apache_edge/vhosts/<name>.conf`.

If `api.culpur.net.conf` is not yet present in the puppet-control vhost
source set (per the apache_edge profile bootstrap note —
`docs/CAPTURE_LIVE_FILES.md` is still TBD), capture the live vhost from
guard FIRST, then apply the `flushpackets=on` patch:

```sh
# Capture from live guard (one-time bootstrap)
scp guard:/etc/apache2/sites-available/api.culpur.net.conf \
    /Users/soulofall/projects/puppet-control/environments/production/site-modules/profile/files/apache_edge/vhosts/api.culpur.net.conf

# Add api.culpur.net.conf to the $vhost_files array in apache_edge.pp
```

`apache_edge.pp`:


## 6. Verify after apply

Use curl from the operator workstation (Authentik cookie required, or
test from dev0001 inside the proxy chain). The key behaviour: SSE
`event:` lines must arrive one-at-a-time, NOT all at once after the
backend closes the stream.

```sh
# From dev0001 (bypasses Authentik for the bench test)
curl -N \
  -H 'Content-Type: application/json' \
  -H 'Accept: text/event-stream' \
  -H 'X-Forwarded-User: chatgpt@culpur.net' \
  -H 'X-Forwarded-Email: chatgpt@culpur.net' \
  -H 'X-Forwarded-Groups: anvilhub-publishers,users' \
  -d '{"prompt":"A weather tool taking a city","stream":true}' \
  http://10.0.70.80:4090/api/builder/spec

# From outside, via the public edge (full chain):
curl -N \
  -b 'authentik_proxy=...' \
  -H 'Content-Type: application/json' \
  -H 'Accept: text/event-stream' \
  -d '{"prompt":"A weather tool taking a city","stream":true}' \
  https://api.culpur.net/api/builder/spec
```

Expected: `event: token` / `data: {...}` lines appear one per token, with
visible delays between them. With buffering still in place (the bug),
nothing arrives until the final `event: done` and the whole stream
emerges at once.

The `-N` flag (`--no-buffer`) is required on the curl side — curl itself
buffers stdout by default and would mask the underlying behaviour.

---

## 7. Rollback

If `flushpackets=on` breaks something unexpected (it should not — it is
the canonical SSE proxy setting), revert the vhost file change in
puppet-control and re-run Puppet on both edges. The micro-service is
unaffected by this change; only the proxy directive is altered.
