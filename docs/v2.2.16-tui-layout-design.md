# v2.2.16 TUI Layout System — Design Spec

**Status:** Approved-pending-decisions (six open questions in §11)
**Date:** 2026-05-16
**Author:** Maverick + Plan agent
**Target release:** v2.2.16 (headline feature)
**Live preview:** https://anvilhub.culpur.net/tui-preview (six variants A–F)

## Repo facts verified before writing this

- TUI lives in `crates/anvil-cli/src/tui/{mod.rs, layout.rs, state.rs, input_handler.rs, redraw.rs, configure_types.rs, ...}`. `mod.rs` is ~4920 lines and owns the single `AnvilTui::draw` method that hard-codes a 3-zone vertical Layout split.
- `tui::layout` already exists as a *helper* module for cursor math + status-line spans (NOT a layout-trait dispatcher), and `ratatui::layout::Layout` is imported as `Layout` in `mod.rs`. Our new abstraction must be called something different — we use **`TuiLayout`** / **`LayoutMode`**.
- Settings live in `~/.anvil/config.json` (not `settings.json` — the prompt mis-stated this; `settings.json` is a separate Claude-Code-compat file handled by `runtime::config`). The wizard writes via `wizard_save_config()` in `crates/anvil-cli/src/wizard.rs:141`.
- The "8-axis" contract is the eight `SlashCommandSpec` fields enforced by `every_slash_command_variant_has_a_spec` (and the orphan/handler tests around it) in `crates/commands/src/lib.rs:2582`. Adding `/layout` requires: (1) a `SlashCommand::Layout` enum variant, (2) parser arm, (3) `SlashCommandSpec` entry, (4) variant-name arm in the drift test, (5) exemplar in the drift test, (6) handler arm in `HANDLER_NAMES`, (7) dispatch in `dispatch.rs`, (8) handler implementation.
- `Tab` (in `state.rs:439`) already carries all per-tab state. `AnvilTui` (in `mod.rs:90`) already has `RedrawScheduler` (`redraw: redraw::RedrawScheduler`) plus a `redraw_pending` gate. `DirtyRegions` is a bitset we can extend.
- `ConfigureState` (in `configure_types.rs:11`) is already a sum type of screens. `ConfigureAction` enum (in `configure_types.rs:69`) is where new live-actions go; the REPL dispatcher in `main.rs` consumes them.

## 1. Settings schema

### Type definitions

Add to `crates/runtime/src/config/mod.rs` (alongside `WorktreeConfig`, `EgressConfig`, etc.):

```rust
/// v2.2.16: TUI layout selection. Persisted as `tui_layout` in
/// `~/.anvil/config.json`. Each layout has an independent `tabs` flag
/// because tabs are a layout-axis feature (NOT a global toggle) — the
/// six visual variants A/B/C/D/E/F are the cross-product of `kind` and
/// `tabs`.
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub struct TuiLayoutConfig {
    pub kind: TuiLayoutKind,
    pub tabs: bool,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum TuiLayoutKind {
    /// Layout A/D — persistent left rail + swappable right deck.
    VerticalSplit,
    /// Layout B/E — FOCUS / LOG / CONTEXT three-pane vim-modal.
    ThreePane,
    /// Layout C/F — timestamped single-column journal with Ctrl-K palette.
    Journal,
}

impl Default for TuiLayoutConfig {
    fn default() -> Self {
        // Migration: pre-v2.2.16 users had a single-deck + tab-strip UI
        // closest to VerticalSplit{tabs:true}. See §9.
        Self { kind: TuiLayoutKind::VerticalSplit, tabs: true }
    }
}
```

### Serde representation (chosen: tagged object, not enum-with-payload)

```json
{
  "tui_layout": {
    "kind": "vertical-split",
    "tabs": true
  }
}
```

Rationale: a flat `{kind, tabs}` object is friction-free for hand-editing and `/configure layout three-pane --no-tabs` to round-trip. A discriminated `serde(tag="kind")` enum would also work but forces every variant to be re-keyed on the wire; the data is simpler.

Backwards-compat: when `tui_layout` is absent, `TuiLayoutConfig::default()` resolves to **vertical-split + tabs ON**, which is the closest visual match to the current TUI (single right-side content area with a tab strip at the top — see `mod.rs:1219` header rows split).

Also accept a short-form string for ergonomics:

```json
{ "tui_layout": "vertical-split" }       // → {kind: vertical-split, tabs: true}
{ "tui_layout": "vertical-split-tabs" }  // → {kind: vertical-split, tabs: true}
{ "tui_layout": "three-pane" }           // → {kind: three-pane,   tabs: false}
```

The short-form lets the wizard write `"vertical-split"` and lets the `--no-tabs` CLI flag in `/configure` produce a clean diff. Loader code lives in `config/mod.rs` next to `parse_optional_otel_config()`.

### CLI alias mapping table

| Short name (config / CLI) | Variant | Mock |
|---|---|---|
| `vertical-split`         | `{VerticalSplit, tabs:false}` | A |
| `vertical-split-tabs`    | `{VerticalSplit, tabs:true}`  | D |

## 2. Wizard integration

### Sequence placement

The wizard currently has 7 numbered steps in `wizard.rs::run_first_run_wizard` (see `wizard.rs:715` "Step 7 of 7: TUI Preferences" which bundles mouse capture, theme, permission mode). The layout choice is a *bigger* decision than any of those: it changes which keybinds the user must learn. We recommend **promoting it to a top-level step** and renumbering:

- Step 7: TUI Layout  ← NEW
- Step 8: TUI Preferences (mouse / theme / permission mode — was step 7)

Insert the new step **before** mouse/theme/permission because (a) the layout choice shapes whether mouse capture is even useful (Journal layout uses Ctrl-K palette, not clickable tabs) and (b) the permission-mode prompt at the end is the natural "now press enter to begin" moment.

### Direct-path slash command

Two entry points, both feed the same handler:

1. `/configure layout three-pane --no-tabs` — goes through the existing `Configure { args }` parser. Extend the configure handler to dispatch `args == "layout ..."` to the layout sub-handler.
2. `/layout three-pane --no-tabs` — a NEW top-level slash command. Required because the 6-variant axis is a headline feature and deserves its own muscle-memory.

#### Argument grammar

```
/layout                                       → opens the Layout sub-screen of /configure
/layout list                                  → prints all 6 variants with descriptions
/layout <kind> [--tabs | --no-tabs]           → sets and applies immediately
/layout reset                                 → restore default (vertical-split + tabs)

<kind> ∈ vertical-split | three-pane | journal
       | vertical-split-tabs | three-pane-tabs | journal-tabs
```

When `--tabs` / `--no-tabs` is absent and `<kind>` has no `-tabs` suffix, preserve the current `tabs` setting. When suffix and flag conflict (e.g. `vertical-split-tabs --no-tabs`), the flag wins (explicit overrides shorthand).

## 4. Live-switch atomicity

### Goal

`/layout three-pane` mid-session must NOT:

- drop in-flight conversation (per-tab `ConversationRuntime` and its worker thread keep streaming).
- drop typed-but-unsubmitted input (the `Tab::input` String).
- drop the message queue (`Tab::message_queue: VecDeque<String>`).
- drop the per-tab pending permissions modal.
- drop the scrollback ring buffer (`Tab::scrollback`).

It MUST:

- repaint the entire screen at the next frame boundary (drop layout-specific *visual* state only).
- request a full-region redraw so no stale glyphs survive.

### Mechanism

Add a field to `AnvilTui`:

```rust
pub struct AnvilTui {
    // ... existing fields ...
    pub(super) layout: TuiLayoutConfig,
    /// Layout-local visual state. Reset to `LayoutLocalState::for_kind(kind)`
    /// on every live-switch. Shared `AnvilTui`/`Tab` state is untouched.
    pub(super) layout_local: LayoutLocalState,
}
```

Where `LayoutLocalState` is an enum carrying only the fields that DON'T survive a switch:

```rust
pub(super) enum LayoutLocalState {
    VerticalSplit {
        right_deck_mode: RightDeckMode,  // Conversation | Transcript | ToolResults
        rail_selected: usize,
    },
    ThreePane {
        vim_mode: VimMode,               // Normal | Insert | Command
        focus_buffer_id: Option<TabId>,  // which tab fills the FOCUS pane right now
        command_line: String,            // ":q", ":w" etc.
    },
    Journal {
        palette_open: bool,
        palette_query: String,
        palette_selected: usize,
    },
}
```

### Switch protocol (`AnvilTui::set_layout(new: TuiLayoutConfig)`)

1. If `new == self.layout`, no-op (avoid a wasted redraw).
2. Persist to `~/.anvil/config.json` (best-effort; on IO error log and continue).
3. Replace `self.layout = new`.
4. Replace `self.layout_local = LayoutLocalState::for_kind(new.kind)`.
5. `self.redraw.request_full()` — sets force + ALL regions dirty. (`redraw.rs:133`.)
6. `self.request_redraw_reason(RedrawReason::Other)` so the wait-loop gate fires.

Steps 3–6 are inside one `&mut self` call → no concurrent observation. The next iteration of the main/wait loop picks up the change.

### What is shared (survives switch) vs layout-local

| State | Owner | Survives? |
|---|---|---|
| `Tab.log` (conversation history) | `Tab` | ✅ shared |
| `Tab.pending_text` (streaming buffer) | `Tab` | ✅ shared |
| `Tab.input`, `Tab.cursor` | `Tab` | ✅ shared |
| `Tab.message_queue` | `Tab` | ✅ shared |
| `Tab.scrollback` | `Tab` | ✅ shared |
| `Tab.completion` popup | `Tab` | ✅ shared |
| Per-tab `ConversationRuntime` + worker | `LiveCli.tab_runtimes` | ✅ shared (lives outside TUI) |
| `pending_permissions: HashMap<usize, _>` | `AnvilTui` | ✅ shared |
| `agent_rows` | `AnvilTui` | ✅ shared |
| `git_branch`, `git_diff_stats`, tokens, model | `AnvilTui` / `Tab` | ✅ shared |
| `configure_state` (the overlay) | `AnvilTui` | ✅ shared (closed before switch) |
| `theme`, `status_line_config` | `AnvilTui` | ✅ shared |
| Right-deck mode (which deck panel is showing) | `LayoutLocalState::VerticalSplit` | ❌ layout-local |
| Vim modal mode + `:` command line | `LayoutLocalState::ThreePane` | ❌ layout-local |
| Ctrl-K palette open/query | `LayoutLocalState::Journal` | ❌ layout-local |
| `tab_hits` (click geometry) | `AnvilTui` | ❌ re-computed every draw anyway |

### DirtyRegions extension

Add three new flags to `redraw.rs::DirtyRegions` so each layout's panes can request scoped repaints later:

```rust
pub const RAIL: Self    = Self(1 << 7);   // Layout A/D left rail
pub const FOCUS: Self   = Self(1 << 8);   // Layout B/E FOCUS pane
pub const CONTEXT: Self = Self(1 << 9);   // Layout B/E CONTEXT strip
// JOURNAL_BODY reuses SCROLLBACK (single-column → same region)
// BUFFER_LINE / THREAD_SWITCHER reuse TAB_STRIP (same role)
```

(Note: `DirtyRegions(u8)` must widen to `u16`. The `ALL` constant updates accordingly. Existing callers continue to compile because `request(DirtyRegions::INPUT)` is API-stable.)

## 5. Per-layout state model

| State field | A/D Vertical-Split | B/E Three-Pane | C/F Journal |
|---|---|---|---|
| `Tab.log` | rendered in right deck (Conversation mode) | rendered as compact bullets in LOG pane | rendered as timestamped entries in body |
| `Tab.pending_text` | streamed into right deck | streamed into FOCUS pane | appended to current journal entry |
| Active assistant action | derived from `pending_text + log.last()` (shown inline) | **dedicated FOCUS pane** — full-width top third | inline in journal stream |
| `Tab.input` | floating prompt at bottom | hidden in NORMAL mode; visible in INSERT mode | bottom row, always visible |
| `AnvilTui.agent_rows` | "agents" section of left rail | NOT shown (FOCUS is single-action) | rendered as journal entries when active |
| `AnvilTui.pending_permissions[tab]` | yellow ⚠ on tab/rail entry | red banner over CONTEXT strip | inline as `[permission requested]` journal entry |
| Sessions list (sibling tabs) | top of left rail | hidden (FOCUS is single-tab; tabs via E buffer line if tabs ON) | hidden (threads via F anchor line if tabs ON) |
| Model / tokens / cost (status) | bottom status row | **CONTEXT strip** (third pane) | top anchor line + collapsed status row |
| Git branch / diff stats | bottom status row | CONTEXT strip | top anchor line |
| `Tab.completion` popup | inline above input | inline above input (INSERT mode only) | inline above input + Ctrl-K palette |
| `Tab.scrollback` state | drives right deck scroll | drives LOG pane scroll | drives body scroll |
| `transcript_verbose` | toggles right deck Transcript mode verbosity | toggles LOG pane verbosity | toggles tool-call expansion inline |

**Read this row left-to-right**: every cell points to the same backing data; layouts only differ in *where* and *how* they paint it. Therefore live switching is paint-only and never touches model state.

### Trait

```rust
// crates/anvil-cli/src/tui/layouts/mod.rs

use ratatui::Frame;
use crossterm::event::KeyEvent;

/// One implementation per architectural family (A/D, B/E, C/F).
/// Each impl handles BOTH variants by branching on `cfg.tabs` internally —
/// this keeps the dispatch table small and the variant-specific code
/// co-located.
pub(super) trait TuiLayoutRenderer {
    /// Paint one frame for the given snapshot. Pure function of inputs;
    /// no side effects on shared state (mutates only `local` for cursor
    /// position / scrollback follow).
    fn render(&self, frame: &mut Frame, snap: &LayoutSnapshot, local: &mut LayoutLocalState);

    /// Consume one key event. Returns the resulting outcome (Continue,
    /// Submit, Exit, NewTab, ConfigureAction). Layout-local key handling
    /// (vim modal, Ctrl-K palette) happens here; shared keys (Ctrl+C,
    /// Ctrl+T, etc.) are pre-handled by the dispatcher.
    fn handle_key(
        &self,
        key: KeyEvent,
        snap: &mut LayoutSnapshotMut,
        local: &mut LayoutLocalState,
    ) -> EventOutcome;

    /// Layout-specific cursor position (terminal row/col) for the next
    /// frame, so `terminal.draw()` can set it correctly.
    fn cursor_position(&self, snap: &LayoutSnapshot, local: &LayoutLocalState) -> Option<(u16, u16)>;
}
```

`LayoutSnapshot` and `LayoutSnapshotMut` are the existing draw-snapshot pattern in `mod.rs:879-987` extracted into a struct — this keeps the trait callable without `&mut AnvilTui` and prevents layouts from accidentally mutating shared state.

### Dispatcher

```rust
// crates/anvil-cli/src/tui/layouts/mod.rs

pub(super) fn dispatch_render(
    cfg: TuiLayoutConfig,
    frame: &mut Frame,
    snap: &LayoutSnapshot,
    local: &mut LayoutLocalState,
) {
    match cfg.kind {
        TuiLayoutKind::VerticalSplit => vertical_split::Renderer { tabs: cfg.tabs }.render(frame, snap, local),
        TuiLayoutKind::ThreePane    => three_pane::Renderer    { tabs: cfg.tabs }.render(frame, snap, local),
        TuiLayoutKind::Journal      => journal::Renderer       { tabs: cfg.tabs }.render(frame, snap, local),
    }
}
```

### How `AnvilTui::draw` shrinks

Today `mod.rs:799-1955` is one giant `draw()` function. The migration:

1. Keep the snapshot-collection block (lines 879–987) as-is. Refactor into `fn collect_snapshot(&self) -> LayoutSnapshot`.
2. Replace the body of `terminal.draw(|frame| { ... })` (the area-splitting + tab-bar + content rendering, ~lines 1184–1750) with `dispatch_render(self.layout, frame, &snap, &mut self.layout_local)`.

## 7. Tabs strategy per layout

The user explicitly stated tabs are not optional — they're a layout *axis*. Below: what tabs look like in each architecture, and what `tabs: false` means (always: tab strip hidden, multi-tab state preserved internally, single-tab rendering of the active tab).

### Layout A/D — Vertical Split

- **D (tabs ON)**: tab strip rendered **inside the right deck** at the top (matches mockup `https://anvilhub.culpur.net/tui-preview`). Left rail stays anchored across tab switches. Tab strip row 0 of the right deck area; deck content fills rows 1..N. Click handlers from `tab_hits` work as today.
- **A (tabs OFF)**: tab strip suppressed. The right deck claims the full vertical span. Multi-tab state still exists; `Ctrl+T` creates a new tab but the user can only see it via `/tab switch <id>` or by enabling tabs again. (Decision: don't auto-collapse to a single tab — that would discard the runtime. Just hide the chrome.)

### Layout B/E — Three-Pane

- **E (tabs ON)**: a **buffer line** rendered as terminal row 0 above the FOCUS pane. Format: `[1] anvil-dev*  [2] aegis-fix⚠  [3] deploy~  [+]`. Modifier conventions (vim-buffer style):
  - `*` = unread (mirrors today's `has_unread`)
  - `⚠` = pending permission (mirrors `pending_permissions`)
  - `~` = in-flight turn streaming
  - `[+]` = new buffer (same handler as `Ctrl+T`)
  - Active buffer: highlighted with theme accent.
  - Keybinds: `gt` next buffer, `gT` prev buffer, `Ctrl+1..9` jump-to-buffer-N, `:b<id>` switch (ex-cmd mode), `:bd` close buffer.
- **B (tabs OFF)**: buffer line absent. FOCUS pane starts at row 0. Switch via `/tab switch <id>` only.

### Layout C/F — Journal

- **F (tabs ON)**: a single-row **thread switcher anchor line** at terminal row 0: `anvil-dev · aegis-fix · deploy-prep · +`. Active thread underlined or bold with accent. Each thread has its own `model` (read from `Tab.model`). Clicking a name switches (uses the same `tab_hits` infrastructure with different geometry). Keybinds: `Ctrl+Tab` / `Ctrl+Shift+Tab` cycle, `Ctrl+1..9` jump.
- **C (tabs OFF)**: anchor line absent. Journal body starts at row 0. Multi-thread state preserved; `Ctrl+K` palette can still list threads as a command (`Switch thread: anvil-dev`).

### Common: `Ctrl+T` (new tab) and `Ctrl+W` (close tab)

These remain global, regardless of layout or `tabs` setting — they exist because `ConversationRuntime` instancing is independent of the visual axis. When `tabs:false`, the new tab is created but not visible (status line shows `[N tabs]`).

## 9. Migration path

### v2.2.15 → v2.2.16 upgrade behaviour

1. User upgrades binary.
2. First launch: `~/.anvil/config.json` has no `tui_layout` field.
3. Loader returns `TuiLayoutConfig::default()` = `{kind: VerticalSplit, tabs: true}` (Layout D — the closest visual match to current).
4. **A one-time toast** appears in the active tab's `LogEntry::System` on the first frame:
   > `◆ Anvil v2.2.16 introduces TUI layouts. You're on Vertical Split + Tabs (current behaviour). Try /layout list — six variants. /layout three-pane is a new vim-modal view worth trying.`
5. The toast is suppressed forever by writing `"tui_layout_intro_seen": true` to `config.json` on first display.

### Wizard behaviour for upgraders

If `config.json` exists but `tui_layout` is absent AND the user is in an interactive terminal AND the toast was just shown, that's it — no re-prompt. The wizard only runs when `config.json` is missing entirely (`anvil_config_json_exists() == false`, `wizard.rs:24`). The toast handles the announcement.

### State migration

None required. All migration is *config-shape only* — no on-disk session data changes. The `Tab` struct, `ConversationRuntime`, scrollback, history, etc. are unchanged.

### Rollback

Setting `"tui_layout": "vertical-split-tabs"` (or omitting the key entirely) restores pre-v2.2.16 behaviour pixel-for-pixel because Layout D's renderer is the existing draw code, refactored not rewritten.

## 11. Risks + open questions

### Risks

1. **`mod.rs` extraction is mechanical but huge.** `AnvilTui::draw` is ~1100 lines in one function (lines 799–1955). Splitting it into per-layout renderers risks accidentally changing the existing render. **Mitigation**: golden-snapshot the current Layout D rendering FIRST (before extraction), then refactor until snapshots match exactly.
2. **`DirtyRegions` widens from `u8` to `u16`.** Anything reading the raw `.0` field externally breaks. **Mitigation**: nothing in tree reads it directly (verified `grep -rn 'DirtyRegions(0'` — only the const definitions). The `pub const fn is_empty/contains/union` API is stable.
3. **Vim modal in Layout B conflicts with global `Ctrl+T` / `Ctrl+W`.** A vim purist would expect `:bn` to be the only way; we keep both. **Mitigation**: documented in the keybind map; tests cover both code paths.
4. **Mouse capture vs vim modal interaction.** Layout B/E with mouse capture on means clicks scroll panes — but vim users expect `j/k`. **Mitigation**: in Layout B/E, mouse capture is treated as scroll-only; click-to-switch-tab still works in E because the buffer line is at row 0 (we record `tab_hits` for it).
5. **Web viewer (`relay_tx`) is layout-agnostic.** The relay sends raw events; the web viewer renders its own layout. **Risk**: if a layout displays something the web viewer can't render, parity breaks. **Mitigation**: layouts are LOCAL TUI rendering only; no relay payload changes. `web_available: false` on the `/layout` spec (web viewer can't switch the TUI's local layout).
6. **First-run wizard step count is now 8.** Tests that assert "Step 7 of 7" will break. **Mitigation**: grep for "Step 7" in wizard tests during implementation; update.

### Decisions locked (2026-05-16)

User answered all 7 open questions:

1. **Settings file: `~/.anvil/config.json`** (consistent with `theme`, `tui_mouse_capture`, `permission_mode`).
2. **Upgrade UX: one-time toast** in `LogEntry::System` (not a mini-wizard).
3. **`/configure layout` placement: top-level Configure entry** (not nested under Display).
4. **Vim mode in Layout B: `Esc` DISCARDS the draft.** True vim semantics. Draft text is wiped on Esc; `i` re-enters insert mode to a blank buffer. The only way to save a draft mid-typing is `:w` (command mode), which writes it to `Tab.input` for later resume. Drafts are explicit, not implicit.
5. **Ctrl-K palette: full VS Code / Linear scope** — slash commands + threads + recent prompts + files. Requires fuzzy index over `Tab.history` and `file_cache`.
6. **Right-deck modes (Layout A/D): three as proposed** — Conversation, Transcript, ToolResults, cycled by `Ctrl+R`.
7. **Tabs OFF in Layout C: preserve multi-thread state internally.** Hide thread switcher chrome, but all `Tab`s remain alive. Ctrl+K palette can still switch threads via "Switch thread: anvil-dev" command. No data loss.
