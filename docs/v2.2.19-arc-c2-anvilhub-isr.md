# v2.2.19 Arc C2 — AnvilHub /about page ISR migration

**Status:** spec-only (anvilhub-web is not checked out on the local Mac; this
doc describes the exact diff to apply on dev0001 at `/home/anvilhub/anvilhub-web`).

**Linked task:** #714 C2

## Problem (v2.2.18 post-mortem)

The AnvilHub `/about` page was previously set to `dynamic = 'force-dynamic'`
to avoid serving a stale changelog after each release. That setting forces
Next.js to render the page on every request — but it has **no effect on
Cloudflare's edge cache**, which still served the v2.2.17-era page for ~1 hour
after the v2.2.18 deploy. Three users hit stale data during that window before
we manually purged the CF cache via the dashboard.

`force-dynamic` solves the wrong layer of the cache problem. It defeats
Next.js's data cache but leaves CF caching the resulting HTML response. The
correct fix is two-layer:

1. **Next.js**: use ISR with a short `revalidate` window (60 s) so the page
   regenerates on demand within a minute of a deploy.
2. **Cloudflare**: send `s-maxage=60` on the `/about` response so CF respects
   the same TTL.
3. **Active invalidation**: on every release, the anvil-release MCP hits an
   authenticated `/api/revalidate` route that calls Next.js's
   `res.revalidate('/about')` to force-rebuild the page immediately,
   short-circuiting the 60 s window.

This gives a worst-case staleness of:
- ≤ 60 s if the revalidation webhook fails or is skipped.
- ≤ 5 s (CDN propagation) if the webhook fires successfully.

If the page does any data fetching, ensure each `fetch()` call respects the
ISR window — Next.js inherits `revalidate` automatically for fetches without
explicit `cache: 'no-store'`. If any fetch is force-disabled with
`{ cache: 'no-store' }`, change it to `{ next: { revalidate: 60 } }`.

### 2. New file `app/api/revalidate/route.ts`

```ts
import { NextRequest, NextResponse } from 'next/server';
import { revalidatePath } from 'next/cache';

// On-demand revalidation endpoint hit by the anvil-release MCP pages-update
// phase. Requires a shared secret in the env (REVALIDATE_SECRET) — do NOT
// hardcode. Without the secret, the route returns 401 to prevent third
// parties from purging our cache on demand.
//
// Usage from the release pipeline:
//   curl -sf "https://anvilhub.culpur.net/api/revalidate?path=/about&secret=$SECRET"
//
// Response:
//   200 { revalidated: true, path: '/about' }
//   400 { revalidated: false, error: 'path query param required' }
//   401 { revalidated: false, error: 'invalid secret' }
//   500 { revalidated: false, error: '<exception message>' }

export async function POST(req: NextRequest) {
  return GET(req);
}

export async function GET(req: NextRequest) {
  const params = new URL(req.url).searchParams;
  const secret = params.get('secret');
  const path = params.get('path');

  if (!process.env.REVALIDATE_SECRET) {
    return NextResponse.json(
      { revalidated: false, error: 'REVALIDATE_SECRET not configured on server' },
      { status: 500 },
    );
  }

  if (secret !== process.env.REVALIDATE_SECRET) {
    return NextResponse.json(
      { revalidated: false, error: 'invalid secret' },
      { status: 401 },
    );
  }

  if (!path) {
    return NextResponse.json(
      { revalidated: false, error: 'path query param required' },
      { status: 400 },
    );
  }

  try {
    revalidatePath(path);
    return NextResponse.json({ revalidated: true, path });
  } catch (err: unknown) {
    const msg = err instanceof Error ? err.message : String(err);
    return NextResponse.json(
      { revalidated: false, error: msg },
      { status: 500 },
    );
  }
}
```

### 3. CF `s-maxage` header

Two options, pick the one consistent with the rest of `/about`'s response shape:

**Option A — middleware (preferred if you already have one):**

```ts
// middleware.ts
import { NextResponse, type NextRequest } from 'next/server';

export function middleware(req: NextRequest) {
  const res = NextResponse.next();
  if (req.nextUrl.pathname === '/about') {
    res.headers.set('Cache-Control', 'public, s-maxage=60, stale-while-revalidate=300');
  }
  return res;
}

export const config = { matcher: ['/about'] };
```

**Option B — per-route export (if no middleware exists):**

In `app/about/page.tsx`, add a `headers()` call in a Server Component layout
or return the response from a Route Handler. Simpler: middleware is the
right home for this.

The `stale-while-revalidate=300` lets CF serve a slightly stale response while
asynchronously re-fetching, eliminating user-visible latency on the 60 s
boundary.

### 4. Env var on dev0001

Add to `/home/anvilhub/anvilhub-web/.env.production` (or whatever the existing

Restart with `pm2 restart anvilhub-web` after the change.

Stash the secret in the vault under `anvilhub/revalidate-secret` so the
anvil-release MCP can pick it up via `env_or_vault('ANVILHUB_REVALIDATE_SECRET')`.

### 5. anvil-release MCP — fire the webhook after pages-update

In the anvil-release MCP's pages-update phase (mcp-servers/anvil-release/lib/
or wherever `anvilhub_update_pages` lives), after a successful page update:

```ts
const secret = process.env.ANVILHUB_REVALIDATE_SECRET;
if (secret) {
  const url = `https://anvilhub.culpur.net/api/revalidate?path=/about&secret=${encodeURIComponent(secret)}`;
  const r = await fetch(url, { method: 'POST' });
  const body = await r.json().catch(() => ({}));
  if (!r.ok || !body.revalidated) {
    console.warn(`[anvilhub] revalidate webhook failed (status=${r.status}): ${JSON.stringify(body)}`);
  } else {
    console.log(`[anvilhub] /about revalidated`);
  }
}
```

If the secret env var is missing, log a warning but don't fail the release —
the 60 s ISR fallback still protects users.

## Verification

After deploy, dry-run the revalidation cycle:

1. `curl -I https://anvilhub.culpur.net/about` — confirm `Cache-Control`
   includes `s-maxage=60`.
2. Edit a visible string in the /about page source, push, build, restart.
3. Hit `/api/revalidate?path=/about&secret=...` — should return
   `{ "revalidated": true }`.
4. `curl -s https://anvilhub.culpur.net/about | grep <visible-string>` — the
   new string should appear within 5 s of the webhook.

## Future-proofing

The same revalidate route can be reused for any other AnvilHub page that
needs on-deploy invalidation. Add new paths by hitting
`?path=/products&secret=...`, etc. The route is generic.

If we ever migrate AnvilHub to the App Router's `revalidateTag` API,
replace `revalidatePath` with `revalidateTag` and key cached fetches by tag.
