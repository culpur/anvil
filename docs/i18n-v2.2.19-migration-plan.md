## Anvil — 6 phases

### Phase A1 — Settings + module scaffold (low risk)

**Files affected:**
- `crates/anvil-cli/src/i18n/mod.rs` *(new, ~40 LOC)* — wraps `rust_i18n` with `current_locale()`, `set_and_persist(lang)`, `available()`, `native_name(code)`.
- `crates/anvil-cli/src/i18n/meta.rs` *(new, ~30 LOC)* — `LANGUAGE_LABELS` native-name table (Aegis-shape).
- `crates/runtime/src/settings.rs` or wherever `~/.anvil/settings.json` is read/written — add `pub locale: Option<String>`.
- `crates/anvil-cli/src/main.rs` — call `i18n::init_from_settings()` at startup, BEFORE the first TUI render or wizard prompt.

**Estimated LOC change:** +120 LOC, ~6 edits.
**Tests:** Unit test `i18n::set_and_persist` writes settings round-trip; `current_locale` falls back to `en` when settings missing.
**Breakage risk:** Low — no UI changes yet.

### Phase A2 — Extract wizard strings (highest user impact)

**Why first:** First-run wizard is the user's first contact surface; per user priority.

**Files affected:**
- `crates/anvil-cli/src/wizard.rs` (4 141 LOC, ~337 string density)
- `crates/anvil-cli/src/wizard_runner.rs`, `wizard_modals.md`(prose only), `wizard_gaps.rs`, `wizard_ollama.rs`, `wizard_qmd.rs`
- `locales/en.yml` — add `wizard.*` keyspace (~150 new keys)

**Replacement pattern:**
```rust
// Before
println!("Welcome to Anvil — your AI coding assistant");
// After (note: must use push_system / tui::log per feedback-tui-stdout-anti-pattern)
push_system(t!("wizard.welcome.title"));
```

**Key namespace plan:**
- `wizard.welcome.*` (title, subtitle, continue_prompt)
- `wizard.providers.*` (label, hint, options)
- `wizard.shell.*`, `wizard.permissions.*`, `wizard.editor.*`, `wizard.ssh.*`, `wizard.summary.*`
- `wizard.errors.*` (validation messages)

**Estimated LOC change:** ~340 edits in wizard.rs, +150 lines in `locales/en.yml`.
**Tests:** Re-run existing wizard tests (insta snapshot tests — these will need re-snapshotting once but should remain stable in en).
**Breakage risk:** Medium — wizard is the most-tested user-facing flow. **Implementer must run `cargo test -p anvil-cli wizard_` after each batch of 50 string extractions, not after the whole pass.**

### Phase A3 — Extract TUI strings (rail, status, modals, command picker)

**Files affected (in priority order, top 5 highest-density first):**
1. `crates/anvil-cli/src/main.rs` (305 density — bulk is CLI help text, top-level errors, REPL prompts)
2. `crates/anvil-cli/src/cmd_static.rs` (243 density — `/status`, `/version`, `/config`, `/memory`, `/help` reports)
3. `crates/anvil-cli/src/cmd_ai.rs` (205 density — assistant-side rendering, error toasts)
4. `crates/anvil-cli/src/tui/mod.rs` + `tui/widgets.rs` + `tui/state.rs` + `tui/modals/` (~250 combined — rail labels, footer hints, modal titles)
5. `crates/anvil-cli/src/configure.rs` (109 density — `/configure` menu)

**Key namespace plan:**
- `cli.help.*`, `cli.error.*` (top-level)
- `tui.rail.*`, `tui.footer.*`, `tui.status.*`, `tui.modal.<name>.*`
- `cmd.<name>.*` for slash command output text (matches existing `locales/en.yml` convention)
- `configure.*` (extend the existing namespace)

**Estimated LOC change:** ~900 edits, +450 lines in en.yml.
**Tests:** `cargo test -p anvil-cli` (snapshot tests for `/status`, `/help`, `/config` will re-snapshot). Add new test `i18n::every_key_in_en_resolves` that loads every key listed in en.yml through `t!()` and asserts no missing-key warnings.
**Breakage risk:** Medium-high. The TUI snapshot suite covers most surfaces but the rail/footer rendering is width-sensitive — German strings may overflow. **Implementer must run the cross-platform width check after Phase A3 closes.**

### Phase A4 — Seed Tier-1 translations (input from translator agent)

This is where the translator agent's work lands. After A2 + A3 close, en.yml has ~600-800 keys. Translator agent receives en.yml as input + the language tier table, returns 7 translated YAML files.

**Files affected:**
- `locales/es.yml`, `locales/zh-CN.yml`, `locales/fr.yml`, `locales/pt-BR.yml` *(new)*, `locales/ru.yml`, `locales/ja.yml`, `locales/de.yml`

**Note:** The 7 stub YAMLs currently in `locales/` cover only the original 80 keys — the translator agent must **extend each to full coverage**, not rewrite.

**Estimated LOC change:** +650 lines × 7 files = ~4 500 lines of YAML.
**Tests:** Test `i18n::every_key_resolves_in_every_locale` — loads each locale, walks every key in en.yml, asserts no missing keys (fall-back-to-en is acceptable runtime behavior, but **CI gate fails if any key is missing**, to force translator coverage).
**Breakage risk:** Low to source code; high to translation quality. Recommend a "review pass" where a native speaker (or another LLM pass with a translation-review prompt) checks 10% of each non-en file.

### Phase A5 — Wire `/language <code>` end-to-end + `--lang` CLI flag

The slash command and locale persistence are already in `utils.rs:323`. This phase makes it actually do something user-visible (because now the strings respond).

**Files affected:**
- `crates/anvil-cli/src/cmd_static.rs::run_language_command` — wire validation against `i18n::available()`, persist to settings via Phase-A1 module.
- `crates/anvil-cli/src/main.rs` — add `--lang <code>` CLI flag; if set, override settings for this invocation only.
- `locales/en.yml` — update the existing `cmd.language.*` and `language.*` keys to list new Tier-1 codes.

**Estimated LOC change:** ~50 LOC.
**Tests:** Integration test: spawn `anvil --lang es /status`, assert output contains Spanish strings.
**Breakage risk:** Low.

### Phase A6 — Language picker step in wizard + `/configure` Language menu

**Files affected:**
- `crates/anvil-cli/src/wizard.rs` — add new wizard step after welcome, before provider picker: "Select your language" with native-name list.
- `crates/anvil-cli/src/configure.rs` — extend the existing `Language` entry (`locales/en.yml:configure.language`) to be a real picker, not a placeholder.

**Estimated LOC change:** ~80 LOC.
**Tests:** Wizard integration test selecting `es`, asserting settings.json `locale: "es"` post-wizard.
**Breakage risk:** Low.

## Cross-cutting: what we CAN'T or SHOULDN'T translate

Documented for both surfaces — implementer must NOT extract any of these into `t!()` / `t()` calls.

1. **Slash command names:** `/model`, `/tab`, `/ssh`, `/configure`, `/language`, etc. stay literal. (Users muscle-memory them; translating them breaks documentation cross-reference.) **Slash command descriptions** ARE translated; the names themselves are not.
2. **Code identifiers + error codes:** `E_VAULT_LOCKED`, `EBADF`, `Result<T, E>` types, JSON field names. Error code prefix stays literal; the *human-readable explanation* attached to it is translated.
3. **Brand names:** Anvil, AnvilHub, Culpur Defense, CC, MCP, OAuth provider names (Anthropic / OpenAI / Ollama / Gemini), passage, BEMA. Never localized.
4. **File paths and config keys:** `~/.anvil/settings.json`, `ANVIL_HOME`, `permissions.mode`. Stay literal.
5. **Upstream provider API response strings:** Anthropic's rate-limit message, OpenAI's content-filter rejection text. We render them verbatim — out of our control.
6. **`cargo` / `rustc` / `git` toolchain output:** Compile errors, git status output. English-only by ecosystem convention; the implementer should NOT attempt to translate captured-and-rendered subprocess output.
7. **Log lines + telemetry events:** Stay English for grep-ability across global ops (per Culpur ops convention). Only the *user-facing* surfaces translate; the OTel spans, `~/.anvil/anvil.log` entries, and structured event names remain en.

This boundary list also covers the cost-chip labels in viewer.html (FREE / OAUTH / LOCAL / API / METERED / CLOUD / UNKNOWN) — they're effectively brand-class tokens for the cost model and stay English.

## Gates before declaring v2.2.19 i18n complete

1. Every key in `locales/en.yml` resolves in every Tier-1 locale (CI gate).
2. `anvil --lang <code> /status` renders entirely in `<code>` for all 7 Tier-1 codes (integration test).
3. viewer.html loads in en, fr, ja, zh-CN, de without console errors and with no English text leaking through (manual gate, screenshots in PR).
4. Wizard first-run flow completes successfully with `--lang de` set (the German strings don't overflow the rail at 80-column width — implementer must verify per `feedback-cross-platform-ux-defaults.md`).
5. `cargo test -p anvil-cli` green; no snapshot drift beyond expected i18n-related re-snaps.
6. release-surfaces.yaml verifier still green (i18n changes don't break the surface manifest).
7. RELEASE-NOTES-v2.2.19.md drafted with i18n as the headline feature, native-name list inline, CC-only naming throughout.
