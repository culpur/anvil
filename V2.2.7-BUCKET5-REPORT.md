# Anvil v2.2.7 — Bucket 5: Ollama Tool-Use Bug Fix

**Date:** 2026-04-20
**Owner:** Backend Expert agent
**Status:** Complete — build green, all tests pass

---

## Root Cause

**Primary failure point: `crates/api/src/providers/openai_compat.rs:normalize_response` (line ~789)**

When Anvil uses Ollama, the request flow is:

```
User prompt
  → DefaultRuntimeClient::stream()          (providers.rs:670)
  → OllamaClient::stream_message()          (ollama.rs:90)
  → OpenAiCompatClient::stream_message()    (openai_compat.rs:185)
  → Ollama /v1/chat/completions SSE stream
  → StreamState::ingest_chunk()             (openai_compat.rs:348)
    ↳ only processes choice.delta.tool_calls — structured only
  → AssistantEvent events emitted to DefaultRuntimeClient
  → turn_executor::build_assistant_message() (conversation/turn_executor.rs:95)
    ↳ ToolUse events become ContentBlock::ToolUse
    ↳ if empty → no tool dispatch
```

When the Ollama model emits tool intent as prose text, XML tags, or JSON
fences instead of structured `tool_calls` SSE chunks:

- `StreamState::ingest_chunk` sees only `choice.delta.content` text — no tool_calls.
- All text goes into `AssistantEvent::TextDelta` events.
- `build_assistant_message` produces only `ContentBlock::Text` — no `ContentBlock::ToolUse`.
- `turn_executor.rs:67` — `pending_tool_uses.is_empty()` is true → loop breaks.
- **No tool is executed.** The model's prose ("I've created the file") is displayed verbatim.
- User sees a false success message.

The non-streaming fallback (`normalize_response`) had the same gap: it only
read `choice.message.tool_calls` and never scanned the text content.

## Changes Made

### 1. New file: `crates/api/src/providers/ollama_tool_parser.rs`

Multi-format Ollama tool-call parser. Public surface:

```rust
pub fn parse_ollama_text_for_tool_calls(
    text: &str,
    had_structured_tool_calls: bool,
) -> OllamaParseResult
```

Handles four alternate formats:

| Format | Strategy |
|---|---|
| Format 2: `<tool_use>…</tool_use>` | `parse_xml_tool_use_tags` — scans for XML tags, extracts JSON body |
| Format 3: JSON in ` ```json ` or ` ``` ` fence | `parse_json_fences` — finds fences, tries JSON parse |
| Format 4: Prose + code block | `parse_natural_language` — matches write-intent phrases, extracts backtick path + code block |
| Fail-loud: prose claimed write | `text_describes_file_write` — detects "I wrote/saved/created the file" without any tool call |

The parser is guarded by `KNOWN_TOOLS` — only tool names in Anvil's built-in
registry are accepted, preventing false positives from arbitrary JSON blobs.

### 2. Modified: `crates/api/src/providers/openai_compat.rs`

`normalize_response` (non-streaming path) now:

1. Runs primary `tool_calls` extraction as before.
2. If there is text content, calls `parse_ollama_text_for_tool_calls`.
3. Extracted inline tool calls are added as `OutputContentBlock::ToolUse`.
4. If a silent-write is detected but no tool call found anywhere, a warning
   `OutputContentBlock::Text` is appended with the fail-loud message.

### 3. Modified: `crates/anvil-cli/src/providers.rs`

`DefaultRuntimeClient::stream` (streaming path) now:

After the SSE stream loop closes:

1. Collects all accumulated text from `AssistantEvent::TextDelta` events.
2. Calls `parse_ollama_text_for_tool_calls`.
3. For each extracted inline tool call, emits a matching `AssistantEvent::ToolUse`
   and displays a tool-call block in the UI (TUI or stdout).
4. If a silent-write is detected, emits the warning as an additional
   `AssistantEvent::TextDelta` displayed in the chat.

### 4. Module exports: `crates/api/src/lib.rs` and `providers/mod.rs`

`ollama_tool_parser` is declared as a public submodule and its types
(`OllamaParseResult`, `ExtractedToolCall`, `ParseSource`, `SilentWriteDetection`)
plus functions (`parse_ollama_text_for_tool_calls`, `silent_write_warning`) are
re-exported from the `api` crate root.

## Fail-Loud Warning

When a model response contains phrases like "I wrote the file", "I saved the
file", "I've created the file" but no tool call was executed anywhere (not
structured, not inline), Anvil emits:

```
WARNING: The model described writing a file but no tool call was executed.
Claimed action: "I've created the file at /tmp/result.txt…"
This often happens with Ollama models that do not fully support
structured tool calls. The file was NOT saved to disk.
```

This message is shown in both TUI mode and plain-text mode and is appended to
the conversation so the model sees it on the next turn.

## Model Compatibility Matrix

See `V2.2.7-BUCKET5-OLLAMA-MATRIX.md` for the full per-model breakdown.

Short summary:
- **Full structured tool_calls:** Qwen 2.5/3, Mistral, Llama 3.2+ (full quant)
- **Inline XML (`<tool_use>`):** CodeLlama, some Llama 3.1, Qwen-Coder
- **Inline JSON fence:** DeepSeek-Coder, Phi 3/4, CodeLlama
- **Prose + code block:** All prose-only models with a code block
- **Fail-loud only:** DeepSeek-R1, Gemma 2, Vicuna, Solar (no tool training)
