# Anvil v2.2.7 Bucket 3 Report — TUI Windows Scrollback & Selection

## Summary

Fixed two TUI bugs affecting Windows (and cross-platform) usability:

- **Bug A**: Text selection broken — ratatui consumed all mouse drag events, blocking the terminal emulator's native Shift+drag selection.
- **Bug B**: No scrollback — the alternate screen buffer discards history; users had no way to see older output.

---

## Changes

### New file: `crates/anvil-cli/src/tui/scrollback.rs`

Lines 1–325 (entire file).

**Why**: Centralises the ring-buffer logic and navigation state machine in one testable unit. Contains:

- `ScrollbackBuffer` — `VecDeque`-backed ring buffer capped at `DEFAULT_CAPACITY` (10,000) lines. Exposes `push`, `len`, `lines_in_range`, and `live_anchor`.
- `ScrollbackState` — `Option<usize>` newtype. `None` = live view; `Some(anchor)` = historical view at that line index. Methods: `page_up`, `page_down`, `go_live`, `effective_anchor`, `is_live`.

### `crates/anvil-cli/src/tui/mod.rs`

**Module declaration** (line ~14): added `pub(super) mod scrollback;`.

**`AnvilTui` struct** (line ~119): removed dead fields `content_scroll_offset` and `content_auto_scroll` (were `#[allow(dead_code)]` stubs never wired up).

**`AnvilTui::new()`** (line ~166): 
- Added platform selection hint as a `LogEntry::System` message on the initial tab at startup.
- Removed `content_scroll_offset: 0` and `content_auto_scroll: true` from struct init.

**`draw()`** (line ~348):
- Before the `terminal.draw()` closure: iterates the current log + pending text, converts each `LogEntry` to its plain-text spans, and pushes new lines into `tab.scrollback` (watermark-based, only appends).
- Snapshots `scrollback_state` and `scrollback_is_live` so the closure can use them without borrowing `self`.
- Pre-fetches `scrollback_view_lines: Option<Vec<String>>` using `scrollback.lines_in_range()` before the closure for historical view.
- Inside closure: when `scrollback_view_lines.is_some()`, replaces the live content lines with the historical buffer lines prefixed by a yellow `─── HISTORICAL VIEW  (Press End to return to live) ───` banner.

**`scroll_to_bottom()`** (line ~1210): also resets `tab.scrollback_state` to `ScrollbackState::live()`.

**`scroll_up(n)`** (line ~1215): instead of only adjusting `tab.scroll`, calls `tab.scrollback_state.page_up()`. First PageUp from live view enters historical mode.

**`scroll_down(n)`** (line ~1221): calls `tab.scrollback_state.page_down()`. When it returns live, resets `tab.scroll = usize::MAX`.

**`scroll_to_live()`** (line ~1257): new method, delegates to `scroll_to_bottom()`. Called by `End` key.

### `crates/anvil-cli/src/tui/input_handler.rs`

**Mouse handler in `read_input()`** (line ~40):

**Bug A fix**: Added check before the `match mouse.kind` block:

```rust
if matches!(mouse.kind, MouseEventKind::Drag(_))
    && mouse.modifiers.contains(KeyModifiers::SHIFT)
{
    // Pass through — do not consume.
} else {
    match mouse.kind { ... }
}
```

When the user holds Shift and drags, the event is *not consumed*. Terminal emulators (Windows Terminal, ConEmu, most Linux VTEs, iTerm2) intercept Shift+drag for native text selection only when the application does not claim the event. By falling through, native selection works.

**Bug B fix — `End` key** in `handle_key()` (line ~113):

```rust
KeyCode::End => {
    if !self.active_tab().scrollback_state.is_live() {
        self.scroll_to_live();
    } else {
        self.cursor_end();
    }
}
```

When in historical scrollback view, `End` returns to live. When already live, it moves the input cursor to end of line (unchanged behaviour).

`PageUp` / `PageDown` already called `scroll_up` / `scroll_down`; those methods now drive the scrollback buffer (no key binding change needed).

## Known Limitations

- **cmd.exe (legacy Windows console)**: Does not reliably report the `SHIFT` modifier on mouse drag events due to its limited VT/mouse protocol implementation. The Shift+drag pass-through will silently do nothing on cmd.exe. Users should upgrade to **Windows Terminal** (free, from the Microsoft Store) which correctly reports modifiers.

- **Scrollback line count is approximate** on wide content: the buffer stores plain text extracted from `LogEntry::to_lines()` using a terminal-width approximation. Lines that wrap differently at the actual render width may show slightly different wrapping in historical view vs. live view. This is a cosmetic issue only.

- **Scrollback is not persisted**: the ring buffer is in-memory and is lost when the session exits. This matches the behavior of terminal emulator scrollback.

- **Historical view uses approximate height**: the `page_up`/`page_down` methods use a hardcoded `est_height = 20` for anchor calculation. The true visible height is only known inside the ratatui draw closure. This means the anchor can be slightly off at the boundary — the `effective_anchor` clamp in `ScrollbackState` compensates for this.
